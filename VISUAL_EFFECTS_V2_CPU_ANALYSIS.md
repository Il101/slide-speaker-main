## 👥 Capacity Planning: Сколько пользователей потянет сервер?

### Сценарий 1: Генерация презентаций

**Предположения:**
- Сервер: 4 CPU cores, 8GB RAM
- Презентация: 15 слайдов
- Время генерации manifest: 1.5 сек
- CPU usage: 1-2% на 1 core
- Memory: 150MB на презентацию

**Capacity:**

```
Последовательная генерация (pessimistic):
──────────────────────────────────────────
1 презентация = 1.5 сек
60 секунд = 40 презентаций
1 час = 2,400 презентаций
1 день = 57,600 презентаций

Параллельная генерация (realistic):
────────────────────────────────────
4 cores × 80% utilization = 3.2 эффективных cores
3.2 cores / 0.02 cores per task = 160 параллельных задач
160 задач × 60 секунд / 1.5 сек = ~6,400 презентаций/час
```

**Bottleneck: Memory, не CPU!**
```
8GB RAM / 150MB per presentation = ~50 одновременных генераций
```

**Реалистичный capacity:**
```
✅ 30-50 одновременных генераций презентаций
✅ 2,000-3,000 презентаций/час
✅ 50,000-70,000 презентаций/день
```

### Сценарий 2: Просмотр презентаций (streaming)

**Предположения:**
- Manifest уже сгенерирован
- Сервер отдаёт JSON (100KB)
- Рендеринг на клиенте

**Нагрузка на сервер:**
```
Для каждого пользователя:
─────────────────────────
🔥 CPU: 0% (просто отдаём файл)
💾 Memory: <1MB (JSON в памяти)
📤 Bandwidth: 100KB/презентацию
```

**Capacity:**
```
Ограничение: Bandwidth + nginx/gunicorn

Типичный сервер:
────────────────
1 Gbps = 125 MB/s
125 MB/s / 0.1 MB per manifest = 1,250 requests/sec
1,250 requests/sec × 3,600 sec = 4,500,000 просмотров/час

Реалистично (с учётом аудио/изображений):
──────────────────────────────────────────
~500-1,000 одновременных зрителей
~10,000-50,000 просмотров/час
```

**Bottleneck: Bandwidth (аудио/изображения), не CPU!**
