# 🔍 Анализ проблем проекта Slide Speaker

## ❌ Критические проблемы найденные и исправленные

### 1. **КРИТИЧЕСКИЙ БАГ: Неправильные отступы в парсерах (ИСПРАВЛЕНО)**

**Файл:** `backend/app/services/sprint1/document_parser.py`

**Проблема:**
- В методах `extract_slides()` как PDF, так и PPTX парсеров код создания объекта `Slide` и добавления его в список находился **ВНЕ цикла** `for`
- Это приводило к тому, что обрабатывался только **последний слайд** из презентации
- Остальные слайды парсились, изображения создавались, но в итоговый манифест попадал только последний

**Локация бага:**
- PDF Parser: строки 440-457 (было вне цикла `for page_num in range(len(doc)):`)
- PPTX Parser: строки 171-186 (было вне цикла `for slide_num, slide in enumerate(prs.slides):`)

**Код ДО исправления (PDF):**
```python
for page_num in range(len(doc)):
    # ... обработка ...
    cues = await self.generate_cues(elements)
    
slide = Slide(  # ❌ ВНЕ ЦИКЛА! (4 пробела вместо 12)
    id=page_num + 1,
    ...
)
slides.append(slide)  # ❌ ВНЕ ЦИКЛА!
```

**Код ПОСЛЕ исправления:**
```python
for page_num in range(len(doc)):
    # ... обработка ...
    cues = await self.generate_cues(elements)
    
    slide = Slide(  # ✅ ВНУТРИ ЦИКЛА (12 пробелов)
        id=page_num + 1,
        ...
    )
    slides.append(slide)  # ✅ ВНУТРИ ЦИКЛА
```

**Исправление:**
- Добавлены правильные отступы (8 дополнительных пробелов)
- Теперь каждый слайд создаётся и добавляется в список внутри цикла
- Все слайды из презентации попадают в манифест

### 2. **Проблема с Docker кэшем и обновлением кода**

**Проблема:**
- При использовании `docker-compose build` с кэшем изменения в коде не всегда попадали в контейнер
- `docker-compose restart` не обновляет код в контейнере, если образ не пересобран

**Решение:**
```bash
# Остановить все контейнеры
docker-compose down

# Пересобрать БЕЗ КЭША
docker-compose build --no-cache backend celery db-init

# Запустить заново
docker-compose up -d
```

**Правильный workflow при изменении кода:**
1. Остановить контейнеры: `docker-compose down`
2. Пересобрать без кэша: `docker-compose build --no-cache [service]`
3. Запустить: `docker-compose up -d`

## 🔧 Системные проблемы проекта

### 3. **Отсутствие автоматической перезагрузки кода в разработке**

**Проблема:**
- Backend и Celery не используют hot-reload должным образом
- Volume mapping не настроен для автоматической перезагрузки изменений

**Рекомендация:**
Добавить volume mapping в `docker-compose.yml`:
```yaml
backend:
  volumes:
    - ./backend:/app  # Mount source code
    - ./.data:/app/.data
```

### 4. **Логирование недостаточно информативно**

**Проблема:**
- В логах парсера не видно, сколько слайдов было обработано ДО сохранения в манифест
- Трудно отследить, где именно теряются данные

**Рекомендация:**
Добавить больше debug логов:
```python
logger.info(f"Extracted {len(slides)} slides from document")
for slide in slides:
    logger.debug(f"Slide {slide.id}: {len(slide.elements)} elements")
```

### 5. **Отсутствие тестов для парсеров**

**Проблема:**
- Нет unit-тестов для проверки корректности парсинга
- Такие критические баги как неправильные отступы могли быть найдены автоматически

**Рекомендация:**
Добавить тесты:
```python
async def test_pdf_parser_multiple_pages():
    parser = PDFParser(Path("test_2_pages.pdf"))
    manifest = await parser.parse()
    assert len(manifest.slides) == 2
    assert manifest.slides[0].id == 1
    assert manifest.slides[1].id == 2
```

### 6. **Смешение синхронного и асинхронного кода**

**Проблема:**
- Celery tasks (синхронные) вызывают async парсеры через `loop.run_until_complete()`
- Это усложняет отладку и может вызывать проблемы с event loop

**Локация:**
`backend/app/tasks.py` строка 73:
```python
manifest = loop.run_until_complete(parser.parse())
```

**Рекомендация:**
- Либо сделать парсеры полностью синхронными
- Либо использовать async Celery tasks (celery-async)

### 7. **Нет валидации данных после парсинга**

**Проблема:**
- После парсинга не проверяется:
  - Соответствует ли количество слайдов в manifest количеству созданных изображений
  - Существуют ли файлы изображений на диске
  - Корректны ли пути в manifest

**Рекомендация:**
Добавить валидацию:
```python
# После парсинга
assert len(manifest.slides) == len(list(slides_dir.glob("*.png")))
for slide in manifest.slides:
    image_path = Path(".data") / lesson_id / "slides" / f"{slide.id:03d}.png"
    assert image_path.exists(), f"Image not found: {image_path}"
```
